## 1. Resumen Ejecutivo

El sistema GVP POS actualmente opera sobre WPF .NET 8 con SQLite, limitado exclusivamente a Windows. La migraciГіn propuesta hacia **Electron + React + TypeScript + TailwindCSS** persigue tres objetivos estratГ©gicos:

1. **Multiplataforma:** Windows, macOS y Linux desde un solo codebase.
2. **Experiencia de usuario moderna:** UI fluida, responsive, con componentes reutilizables.
3. **Mantenibilidad:** El ecosistema React/TypeScript tiene un mercado de talento 10Г— mayor que WPF/XAML.

**Principio rector:** Las capas de negocio (Domain, Application, Infrastructure en .NET) permanecen **intactas**. Se introduce una nueva capa de API REST (ASP.NET Core Minimal API) que expone los servicios existentes. Electron consume esa API. La base de datos SQLite se mantiene.

**InversiГіn estimada:** 4-6 meses hasta MVP. Equipo sugerido: 2 full-stack (React/TypeScript + .NET), 1 QA.

### 3.2 Principios de Arquitectura

1. **Backend .NET intacto:** Las capas Domain, Application e Infrastructure no se modifican en su lГіgica de negocio. Solo se aГұade la nueva capa API y se ajusta `CurrentUserService` para trabajar con JWT en lugar de estado singleton en memoria.

2. **API REST local:** La API corre como proceso hijo de Electron (`localhost:5000`), gestionada por el Main Process. No se expone a la red (bind a `127.0.0.1`).

3. **Frontend stateless:** React no mantiene estado de sesiГіn en memoria mГЎs allГЎ de lo necesario. Cada request incluye el token JWT.

4. **Electron como shell:** El Main Process solo orquesta: lanza/kill el backend .NET, maneja ventanas, actualizaciones, licencias, impresiГіn y acceso a hardware.

```json

  "companyId": "1",
```

#### ConfiguraciГіn JWT (appsettings.json)

```json
{
  "Jwt": {
    "Secret": "<clave-secreta-256-bits-generada-por-instalacion>",
    "Issuer": "SistemaGVP",
    "Audience": "SistemaGVP-Client",

    "RefreshTokenExpirationDays": 7
```

#### ImplementaciГіn

1. **`JwtMiddleware`** вҖ” se ejecuta en cada request autenticado. Extrae el token del header `Authorization`, lo valida, y establece `HttpContext.Items["UserId"]`, `["CompanyId"]`, `["UserRole"]`.

2. **`CurrentUserService` refactorizado:**
   - Cambia de `Singleton` a `Scoped`.
   - Resuelve los datos del usuario desde `IHttpContextAccessor` (claims del JWT).
   - Ya no mantiene `SetCurrentUser()` / `Clear()` con estado mutable.
   - `UserId`, `CompanyId`, `UserName` se obtienen de `HttpContext.Items`.

3. **`AuthService` refactorizado:**
   - `LoginAsync` retorna el token JWT + refresh token.
   - `Logout` invalida el refresh token (blacklist en memoria/cache con TTL).
   - Ya no mantiene `_currentUser` en memoria.

4. **Refresh Token:**
   - Endpoint `POST /api/auth/refresh` вҖ” acepta refresh token, emite nuevo access token.
   - Refresh tokens almacenados en tabla `RefreshTokens` en SQLite (o en cachГ© con persistencia).
   - El cliente (React) detecta 401, intenta refresh, si falla redirige a login.

5. **Almacenamiento en cliente (Electron):**
   - Access token y refresh token se guardan en `electron-store` con `safeStorage` (cifrado por OS).
   - No se almacenan en `localStorage` por seguridad.

### 8.3 Roles y AutorizaciГіn

| Rol | Permisos |
|-----|----------|
| **Admin** | Acceso total: CRUD productos/categorГӯas/clientes/proveedores/usuarios, ventas, inventario, reportes, backups, auditorГӯa, configuraciГіn |
| **Cashier** | Ventas (crear, pausar, reanudar), clientes (CRUD), productos (lectura), inventario (lectura), reportes (lectura) |

La autorizaciГіn se implementa en dos niveles:
1. **Middleware .NET:** `[Authorize(Roles = "Admin")]` en endpoints restringidos.
2. **Frontend React:** El router verifica el rol del usuario desde `authStore` antes de renderizar pantallas admin.

### 11.3 Mejoras Propuestas

1. **Backup programado:** Agregar un timer en el Main Process de Electron que cada N horas (configurable, default 6h) llame a `POST /api/backups` para crear un backup automГЎtico.

2. **RotaciГіn de backups:** Mantener solo los Гәltimos N backups (configurable, default 30). Eliminar los mГЎs antiguos automГЎticamente.

3. **ExportaciГіn a USB/red:** Desde Electron (que tiene acceso al filesystem), permitir copiar un backup a una ubicaciГіn externa (pendrive, carpeta de red).

4. **RestauraciГіn con reinicio:** DespuГ©s de restaurar, el backend necesita reiniciarse para recargar la base de datos. El Main Process lo maneja: mata el proceso .NET, lo reinicia, y notifica al frontend.

5. **Backup pre-cierre:** Opcionalmente, crear un backup automГЎtico al cerrar la aplicaciГіn.

### 13.1 Riesgo CrГӯtico: R1 вҖ” CurrentUserService y AuditorГӯa

**AnГЎlisis detallado:**

El `AppDbContext.SaveChangesAsync` actualmente accede a `_currentUser` (inyectado en el constructor del DbContext). Con JWT, el DbContext se resuelve por request, y `ICurrentUserService` debe inyectarse vГӯa Scoped.

**Plan de mitigaciГіn:**

1. `ICurrentUserService` se registra como **Scoped** en la API.
2. Se implementa `CurrentUserService` que lee claims del `HttpContext.Items` (establecidos por `JwtMiddleware`).
3. `AppDbContext` recibe `ICurrentUserService` por inyecciГіn (ya lo hace).
4. En `SaveChangesAsync`, se usa `_currentUser?.IsAuthenticated == true ? _currentUser.UserId : 0`.
5. Se agregan tests de integraciГіn especГӯficos para verificar que la auditorГӯa registra correctamente userId y companyId con JWT.

### 13.2 Riesgo Significativo: R2 вҖ” Ciclo de Vida Backend

**Plan de mitigaciГіn:**

```
Main Process (Electron) вҖ” backend.ts:

1. Buscar puerto disponible (5000, 5001, 5002...)
2. Spawn: dotnet exec SistemaGVP.API.exe --urls http://127.0.0.1:{port}
3. Health check loop: GET http://127.0.0.1:{port}/health
   - Reintentar cada 500ms, mГЎximo 30 intentos (15 segundos)
   - Si falla, mostrar pantalla de error con opciГіn de reintentar
4. Monitorear stdout/stderr para detectar crashes
5. En app.on('before-quit'), enviar SIGTERM y esperar graceful shutdown
6. Si el proceso muere inesperadamente, mostrar notificaciГіn y reintentar
```

## 14. Decisiones de Arquitectura

### 14.1 ВҝPor quГ© Electron y no Web App PWA?

| Factor | Electron | PWA (Browser) |
|--------|----------|---------------|
| Acceso a hardware (impresora, scanner) | вң… Nativo | вқҢ Limitado |
| Archivos locales (backups, SQLite) | вң… Nativo | вқҢ Limitado |
| Ejecutar proceso .NET local | вң… child_process | вқҢ No posible |
| Modo offline | вң… Nativo | вң… Service Worker |
| Auto-updater | вң… electron-updater | вҡ пёҸ Parcial |
| TamaГұo instalador | вқҢ 130-150 MB | вң… 0 MB |
| InstalaciГіn | вң… Instalador estГЎndar | вң… Visitar URL |

**DecisiГіn:** Electron porque el sistema requiere acceso directo al filesystem, impresoras tГ©rmicas, lectores de cГіdigo de barras, y la capacidad de lanzar el backend .NET como proceso local.

### 14.2 ВҝPor quГ© Minimal API y no Controllers?

- Minimal APIs son mГЎs ligeras, con menos boilerplate.
- El sistema tiene endpoints CRUD simples (no lГіgica compleja de routing).
- Mejor rendimiento (menor overhead).
- MГЎs fГЎciles de mantener en un proyecto de este tamaГұo.

### 14.3 ВҝPor quГ© Zustand y no Redux?

- Zustand es mГЎs simple, menos boilerplate.
- Menor curva de aprendizaje para el equipo.
- Suficiente para el estado global de un POS (auth, cart, UI).
- Excelente integraciГіn con TypeScript.

### 14.4 ВҝPor quГ© shadcn/ui y no Material UI / Ant Design?

- shadcn/ui es headless, da control total sobre el markup.
- Se integra naturalmente con TailwindCSS.
- No impone un design system rГӯgido.
- Componentes copiados al proyecto (no dependencia npm), facilitando personalizaciГіn.
- MГЎs liviano que alternativas (tree-shakeable por naturaleza).

### 14.5 Base de Datos: ВҝMigramos a SQL Server o PostgreSQL?

**DecisiГіn: Mantenemos SQLite.**

- El sistema es single-machine, single-tenant.
- SQLite es cero-configuraciГіn, cero-mantenimiento.
- Los backups son triviales (copia de archivo).
- El rendimiento es mГЎs que suficiente para un POS (miles de transacciones/dГӯa).
- Si en el futuro se necesita multi-sucursal con base centralizada, se puede migrar a PostgreSQL sin cambiar el cГіdigo (EF Core abstrae el provider).
